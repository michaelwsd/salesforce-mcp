Full system design — the platform that replaces Salesforce

human plane AI / agent plane shared backend rebuild / gate

Salesforce & HubSpot retired. The platform runs natively on Azure — its own data (Azure Database for PostgreSQL), its own UI (Next.js), sending/receiving through Microsoft 365 — all inside one Entra ID tenant with AU data residency. Two interfaces sit on one backend: a REST API for the team, an MCP server for Claude.

ESignals & growth detection — the re-approach trigger

How "this GOWT company has grown enough to re-approach" is actually computed. Two layers: a quantitative trigger (Tecala financials vs history) and a qualitative context layer (filtered news). The trigger fires on the numbers; the news explains why.

① Primary trigger — Tecala financials

SourceTecala / Originate (the mandatory gate)
PullFTE · revenue · EBITDA, monthly
Methodsnapshot → Postgres → delta vs historical → threshold
AnchorFTE (reliable) · revenue if real · EBITDA = context only (often estimated for private AU cos)

② Context layer — filtered news

Sourceexisting news scraper (or Firmable CSV — compliant alt)
MethodClaude growth-signal filter → tagged events
Rolethe "why" — contract win · funding · partnership. Context, not the trigger
Monthly: Tecala pull + news scrape → snapshot {fte, revenue, ebitda} to Postgres + Claude-filtered news → diff vs history → threshold (e.g. FTE +20% / 2 quarters) → flag re-approach · news attached as the "why"

Two things to lock down

  • Verify Tecala's revenue/EBITDA provenance. If estimated, the delta is noise — anchor the trigger on FTE, treat EBITDA as context.
  • Scraper vs Firmable CSV for news. Own scraper = you control the pipe (ToS/brittleness risk); Firmable CSV = compliant, manual monthly. News is secondary, so either works — choose deliberately.

FMigration — Salesforce → Azure Database for PostgreSQL (one-time)

The existing Salesforce MCP becomes the migration tool — it already reads every object. Org is small: ~70k records, 4 custom objects, 14 files.

✓ Migrates cleanly (data)

  • Accounts · Contacts · Opportunities · OCR (~31k)
  • Tasks / Events / Notes — activity history (~36k)
  • Files (14 ContentDocuments)
  • Relationships — remap 18-char IDs → Postgres keys
  • The fid__c legacy fields — via the existing field map

✗ Rebuilt, not migrated (behaviour)

  • Automation — flows, validation rules, triggers
  • Reports & dashboards (data moves; logic rebuilt)
  • Formula / roll-up fields — re-implement the calc
  • Field history beyond Salesforce's retention — already lost
  • Permissions / sharing → Entra ID + app roles

GBuild phasing & key risks

Wks 1–6Outreach engine, running alongside Salesforce. Value lands early; SF stays the safety net.
Mths 2–4Azure backend + Next.js UI, mirroring SF data. Build the team's working screens.
Mths 4–6+Migrate & cut over. Retire Salesforce only once UI + reporting reach parity.

Key risks to manage

  • Tecala access — the mandatory data gate; blocks even week 1.
  • Email deliverability — per-mailbox cap + reputation; the main operational risk.
  • Scope creep — "origination workflow" (~6 mo) vs "all of Salesforce" (12 mo+). Hold the line.
  • Ownership — you now maintain it; key-person risk on a small team.
